## Dev Notes

### Schema SQL Completo
[Source: docs/architecture.md#schema-do-banco-de-dados]

```sql
-- Migration 001: Tabela de buscas
CREATE TABLE searches (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  pais TEXT NOT NULL,
  estado TEXT NOT NULL,
  cidade TEXT NOT NULL,
  quantidade INTEGER NOT NULL CHECK (quantidade BETWEEN 1 AND 100),
  status TEXT NOT NULL DEFAULT 'PENDENTE'
    CHECK (status IN ('PENDENTE', 'CONCLUÍDA', 'ERRO')),
  total_encontrados INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Migration 002: Tabela de leads
CREATE TABLE leads (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  empresa TEXT NOT NULL,
  telefone TEXT NOT NULL UNIQUE,
  website TEXT,
  cidade TEXT,
  estado TEXT,
  pais TEXT,
  status TEXT NOT NULL DEFAULT 'LOCALIZADOS'
    CHECK (status IN (
      'LOCALIZADOS','PROSPECTAR','PROSPECTADOS',
      'INTERESSE','TRANSFERIDOS','DESCARTADOS'
    )),
  categoria TEXT CHECK (categoria IN ('DOMÉSTICOS','ESPORTIVOS','MISTO')),
  nota SMALLINT CHECK (nota BETWEEN 0 AND 10),
  data_coleta TIMESTAMPTZ DEFAULT NOW(),
  data_resposta TIMESTAMPTZ,
  data_followup TIMESTAMPTZ,
  qtd_reengajamentos SMALLINT DEFAULT 0,
  search_id UUID REFERENCES searches(id),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Índices
CREATE INDEX idx_leads_status ON leads(status);
CREATE INDEX idx_leads_telefone ON leads(telefone);
```

### Dicionário Oficial de Status
[Source: docs/prd.md#dicionário-oficial-de-status]
| Status | Descrição |
|--------|-----------|
| `LOCALIZADOS` | Encontradas pelo Apify, aguardando aprovação |
| `PROSPECTAR` | Aprovadas, aguardando disparo do WhatsApp |
| `PROSPECTADOS` | WhatsApp enviado, follow-ups em curso |
| `INTERESSE` | Qualificado pela IA, aguardando handoff |
| `TRANSFERIDOS` | Handoff ao Closer (Anderson) concluído |
| `DESCARTADOS` | Rejeitado / bot / sem WhatsApp / opt-out |

### Regra de Idempotência
[Source: docs/architecture.md#api-rest]
- O `telefone` em formato E.164 (ex: `5511999998888`) é a chave de identificação única
- O n8n sempre busca e atualiza leads pelo campo `telefone`
- O constraint UNIQUE garante que o mesmo número nunca seja duplicado
